# Multica Daemon のローカル環境制約とトラブルシュート

## 1. PATH走査の仕様
`multica daemon start` (または `restart`) は、起動時にシステムの環境変数 `$PATH` を走査し、`gemini`, `opencode`, `claude` などのコマンドが実在するプロバイダのみをランタイムとしてリッスンします。
もし一部のエージェントが `offline` になる場合、バックグラウンドシェルやサービス起動時のプロファイルに、ツールのパス（例: `npm` グローバルパスや `C:\Users\<user>\bin` など）が正しく通っていないことが原因である可能性が高いです。パスを `$PROFILE` 等に永続化し、正しくパスが通った状態から Daemon を起動する必要があります。

## 2. ディレクトリ・トラストエラーの回避
ローカル環境でエージェントがワークスペースごとに独立したディレクトリを生成し、そこで Headless 起動する際、Gemini エージェントなどのセキュリティ機構によってブロック（Untrusted directory エラー）されることがあります。
これを回避するためには、Daemon プロセスの環境変数（またはシステム環境変数）に `GEMINI_CLI_TRUST_WORKSPACE=true` 等のフラグを仕込んでおく必要があります。

## 3. タスク検知イベントのロストと再発火
エージェントがオフラインの間に、すでに `todo` になったタスクは、Daemon の再起動後のポーリングにおいて「新規タスク」として検知されない場合があります。
その結果、タスクが `todo` のまま一向にアサインされない事象が発生します。この場合は、タスクのステータスを一時的に `backlog` に戻し、すぐに `todo` へ再変更することでイベントを強制発火させ、Daemon にタスクを拾わせることができます。
