# Multica 運用ポリシー

本ドキュメントは、Multica モノレポにおける **PR からマージ、リリース、デプロイ** までの意思決定を揃えるためのチーム合意用のたたき台である。技術的詳細の一次情報は `README.md`、`.github/workflows/`、および `CLAUDE.md` / `AGENTS.md` のコマンド・境界ルールに従う。

---

## 1. 目的と適用範囲

- **目的**: レビュー滞留の削減、本番影響の明確化、リリース経路の一本化（例外は明示的に定義する）。
- **適用**: 本リポジトリ `main` 向けの変更、および本番・セルフホスト利用者向けのリリース作業。フォーク向けの挙動は GitHub 上の分岐（例: `Release` ワークフローは upstream 専用ガード）に従う。

---

## 2. 基本フロー（固定）

次の順序を **デフォルト** とする。PR 内容による分岐は [§4 レーン分け](#4-レーン分け標準--高リスク) で扱い、本節の骨格は変えない。

1. **作業用ブランチ**を `main` から切る（エージェント作業は `cursor/<説明>-<id>` など、チームの命名に合わせる）。
2. **PR を `main` 向けに開く** → [`.github/workflows/ci.yml`](https://github.com/multica-ai/multica/blob/main/.github/workflows/ci.yml) が走る（`push` to `main` および `pull_request` to `main`）。
3. **必須チェックがグリーン**（フロント: `turbo build typecheck test`、バックエンド: `go build` / migrate / `go test`）。チームが追加した Branch protection はそれに従う。
4. **レビュー承認**後、[§5 マージ](#5-レビューとマージ条件) に従ってマージ。
5. **本番相当のアーティファクトを出す変更**は、`main` に載ったうえで **[§6 リリース](#6-リリースとデプロイ)** に従いタグまたは運用手順で配布する。

---

## 3. CI の前提（事実）

| トリガー | 内容（概要） |
|----------|----------------|
| PR / `main` への push | フロントエンド: Node 22、`pnpm`、`turbo build typecheck test`（`@multica/docs` は除外）。バックエンド: Go、`migrate up`、`go test`。サービス: Postgres（pgvector/pg17）、Redis（テスト用）。 |

詳細は `.github/workflows/ci.yml` を参照する。

## 4. レーン分け（標準 / 高リスク）

「すべての PR を同じ厳しさ」にするとレビューが溜まりやすい。「PR ごとに都度相談」だと再現性がない。そこで **2 レーン**に分け、**高リスク条件はパスと責務で機械的に寄せる**。

### 4.1 標準レーン

- **想定**: ドキュメントのみ、UI の文言・軽微スタイル、テスト追加、リファクタ（挙動変更がレビューで明確に限定されるもの）。
- **レビュー**: 原則 **1 名承認**（チーム規模に応じて `CODEOWNERS` 導入時は自動レビュー要求に委ねる）。
- **追加検証環境**: PR ごとの staging は **必須としない**（CI グリーンを満たす）。

### 4.2 高リスクレーン

次の **いずれか** に該当する変更は高リスクとみなす（PR 説明で明示すること）。

- **データベース**: `server` 配下のマイグレーション新規・変更、スキーマ互換性に触れる変更。
- **認証・セキュリティ**: ログイン、セッション、トークン、権限、サインアップ制御。
- **稼働基盤**: Agent / Runtime / Daemon / WebSocket のプロトコルや実行経路に関わる変更。
- **リリースパイプライン**: `.github/workflows/`、Dockerfile、セルフホスト向け Compose の挙動に直結する変更。
- **破壊的変更**: 既存 API の契約変更、クライアント互換性に影響する変更。

**高リスクレーンの追加ルール**

- **レビュー**: 原則 **2 名承認**、うち少なくとも **1 名は該当領域に慣れたメンバー**。
- **検証**: チームが **共有 staging** を持つ場合は、その環境での確認をマージ条件に含める。staging が無い期間は、最低限 **ローカルまたは CI での `make check` 相当**（貢献ガイドのフル検証手順）を PR 本文に記載する。
- **大きな PR**: 可能なら分割する。分割できない場合は PR 本文に「影響範囲」「ロールバック案」を必須とする。

---

## 5. レビューとマージ条件

| 項目 | 標準レーン | 高リスクレーン |
|------|------------|----------------|
| CI | 必須グリーン | 必須グリーン |
| 承認 | 1 | 2（うちドメイン慣れ 1） |
| ドラフト | レビュー依頼前にドラフト解除 | 同上 |
| マージ戦略 | チーム設定に従う（squash 推奨の場合はその旨を Branch protection に寄せる） | 同上 |

**レビュー滞留への運用**

- レビュー待ち PR は **上限を意識**（例: 担当者あたり同時にReview中のPR数）し、新規実装より **滞留解消を優先**する時間帯を週次で確保する。
- PR は **小さく保つ**（1 PR 1 意図）。巨大 PR は高リスクレーン扱いとする。

## 6. リリースとデプロイ

### 6.1 `main` とリリースの関係

- **マージ先の正**は `main`。日々の統合はここに集約する。

### 6.2 バージョンタグと GitHub Release（公式パイプライン）

セマンティックバージョン形式のタグ `vX.Y.Z`（およびサフィックス付き `vX.Y.Z-<suffix>`）の push が [`.github/workflows/release.yml`](https://github.com/multica-ai/multica/blob/main/.github/workflows/release.yml) を起動する。

想定される成果物（概要）:

- Go: GoReleaser による **CLI バイナリ**および Homebrew tap 連携（upstream リポジトリ前提）。
- コンテナ: **GHCR** 上の `multica-backend` / `multica-web` イメージ（マルチアーキテクチャ）。
- Desktop: CI 上では Linux / Windows 向けパッケージのビルド・アップロード。macOS はコメント上、署名・公証の都合で **手動経路**が併存しうる。

**安定版タグ**（サフィックス無し）では `latest` 相当のタグ付けが有効になるロジックが含まれる。詳細はワークフロー内の `is_stable` を参照する。

### 6.3 セルフホスト利用者への反映

- `SELF_HOSTING.md` のとおり、公式は **GHCR のイメージ**を引くフローが基本。タグに対応したイメージをデプロイする。
- 検証用にローカルビルドする場合は `make selfhost-build` 等、ドキュメント記載の手順に従い、**本番と同じタグ運用**を乱さないようドキュメント化する。

### 6.4 Multica Cloud（マネージド）

クラウド側のデプロイパイプラインは本オープンソースリポのワークフローに含まれない場合がある。**プロダクションへの反映**は、チームの内部 runbook に従い、原則として **どの Git SHA / どのイメージタグが走っているか**を追跡可能にする。

### 6.5 CLI とプロダクションの対応（運用ルール）

プロダクションを更新するリリースでは、利用者向け **CLI の整合**を取るため、チーム方針として **パッチバージョンのタグを付与し GitHub Release を作る**ことを推奨する（詳細はルートの `CLAUDE.md` に記載の CLI リリース手順）。例外はリリースノートに理由を書く。

## 7. 検証環境の戦略（段階的）

すべての PR に常設 staging を用意することはコストが高い。次の **段階**で足す。

| 段階 | 内容 |
|------|------|
| **第1段** | **共有 staging を 1 系統**（`main` の直近またはリリース候補タグをデプロイ）。高リスク PR のマージ前確認に使う。 |
| **第2段** | 滞留・UI 回帰が課題になったら、**高リスクまたは大型 PR 限定**で PR プレビュー環境を追加検討する。 |
| **第3段** | デプロイ頻度が上がったら、staging（統合）と preview（PR 個別）の役割を分離する。 |

---

## 8. ホットフィックス

- **原則**: 通常と同じく PR 経由で `main` に戻し、必要なら **パッチタグ**で急いでリリースする。
- **例外**: 本番救済のためだけの直接コミットを許容する場合は、事後 **必ず PR で整形と説明**を伴うことをチームで合意する。

---

## 9. 変更・問い合わせ

- 本ポリシーの改訂は PR で行い、`docs/` 配下のレビューを経る。
- 「この PR はどのレーンか」迷うときは **高リスクに寄せる**か、該当オーナーに一度だけ確認する。
