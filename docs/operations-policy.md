# Multica 運用ポリシー

## 0. 位置づけ（認識の合わせ方）

**Multica は、他社・他案件のプロダクトを開発・保守・運用していくためのプラットフォーム**である。  
本ドキュメントの主語は、**そのプロダクト群に対する日々の仕事（課題の整理、エージェント割当、レビュー、リリースの合意）**とする。

プラットフォーム自身の改善（本リポジトリでの Multica 本体の変更）も不可欠だが、方針判断は **「載せているプロダクトに与える影響」** を先に置く。  
手続き上の型は多くの場面で共通するが、必ずしも二重管理しない——迷ったら **プロダクト側のリスク** を先に評価する。

技術的詳細の一次情報は `README.md`、`.github/workflows/`、および `CLAUDE.md` / `AGENTS.md` に従う。

---

## 1. 目的と適用範囲

- **目的**: 複数プロダクトを並行運用するときの **レビュー滞留の削減**、**顧客影響の明確化**、**リリース経路の合意**。
- **適用**:
  - **プロダクト作業**: Multica 上の Workspace / Issue / Agent に関わる運用（ボード上の日常）。
  - **プラットフォーム作業**: 本リポジトリの `main` 向け変更、および Multica のバイナリ・イメージ・セルフホスト配布に関わるリリース。
- **フォーク**: GitHub 上のワークフロー分岐（例: upstream のみ Release 公開など）に従う。

---

## 2. 二つのレイヤー（混同しない）

| レイヤー | 何を指すか | 典型的な「レビューが溜まる」 |
|----------|------------|------------------------------|
| **プロダクト** | 各案件・各プロダクトの Issue、担当、エージェント実行、顧客向けマイルストーン | Issue のステータス／レビュー待ち、ブロッカー、納期に効く未処理 |
| **プラットフォーム** | Multica 本体の機能・バグ修正・パフォーマンス（本リポの PR） | プラットフォーム向け PR のレビュー滞留、リリース待ち |

このポリシーは **プロダクトレイヤーを中心**に読む。**プラットフォーム**の PR・タグ・イメージは §6 以降で扱う。

---

## 3. プロダクト運用の基本（中心視点）

プラットフォーム上では、次を **デフォルトの型** とする。

1. **Workspace でプロダクト（または案件単位）を分離**し、混線しない状態を保つ。
2. **Issue が単一の意図**を持つように切る（大きすぎる Issue は分割）。ボード上で優先度が見えるようにする。
3. **人・エージェントの割当**は、そのプロダクトのオーナー／担当が明示できる形にする（誰がレビューし、誰が実行するかが曖昧なまま滞留しない）。
4. **ブロッカー**は Issue 上で可視化し、プラットフォーム側の不具合かプロダクト側の要件かを切り分ける。
5. **顧客影響が大きい変更**（本番設定、認証、公開 URL、課金連携など）は、プロダクト側の合意・確認手順を先に満たしてから実施する。

**レビュー滞留への運用（プロダクト側）**

- 担当あたりの **同時進行上限** を意識する。
- 新規着手より、**既にレビュー待ち・確認待ちの解消**を優先する時間を週次で確保する。
- プラットフォーム本体に手を入れる変更が必要になったときは、**どのプロダクトのどの期限に効くか**を Issue / PR 説明に書く。

## 4. プラットフォーム本体を変更するとき（PR・レーン）

本リポジトリで Multica を直接変更する場合の型である。**プロダクトの Issue 運用と連動**させる（例: 「この修正がないと案件 X がブロックされる」）。

### 4.1 基本フロー（固定）

1. 作業ブランチを `main` から切る（チーム命名に従う）。
2. PR を `main` 向けに開く → [`.github/workflows/ci.yml`](https://github.com/multica-ai/multica/blob/main/.github/workflows/ci.yml) が走る。
3. 必須チェックがグリーン（フロント: `turbo build typecheck test`、バックエンド: `go build` / migrate / `go test`）。
4. レビュー承認後、§4.3 に従ってマージ。
5. 利用者・セルフホストへ反映する変更は §6 に従う。

### 4.2 レーン分け（標準 / 高リスク）

「すべての PR を同じ厳しさ」にするとレビューが溜まりやすい。そこで **2 レーン**に分け、高リスクはパスと責務で寄せる。

**標準レーン**

- **想定**: ドキュメントのみ、軽微 UI、テスト追加、挙動が限定的なリファクタ。
- **レビュー**: 原則 **1 名承認**。
- **追加検証環境**: PR ごとの staging は必須としない。

**高リスクレーン**（いずれか該当時は PR 本文で明示）

- **データベース**: マイグレーション、スキーマ互換性。
- **認証・セキュリティ**: ログイン、セッション、権限。
- **稼働基盤**: Agent / Runtime / Daemon / WebSocket の実行経路。
- **配布・運用**: `.github/workflows/`、Dockerfile、Compose の本番相当の挙動。
- **破壊的変更**: API 契約、クライアント互換性。

**高リスクの追加ルール**

- レビュー: 原則 **2 名承認**（うち **1 名は該当領域に慣れたメンバー**）。
- 検証: 共有 staging がある場合はマージ条件に含める。無い期間は PR 本文に **`make check` 相当**の実施結果を記載する。
- 巨大 PR は分割。分割不可なら「影響範囲」「ロールバック案」を必須とする。

### 4.3 マージ条件（プラットフォーム PR）

| 項目 | 標準レーン | 高リスクレーン |
|------|------------|----------------|
| CI | 必須グリーン | 必須グリーン |
| 承認 | 1 | 2（うちドメイン慣れ 1） |
| ドラフト | 依頼前に解除 | 同上 |

## 5. CI の前提（プラットフォーム／事実）

| トリガー | 内容（概要） |
|----------|----------------|
| PR / `main` への push | フロント: Node 22、`pnpm`、`turbo build typecheck test`（`@multica/docs` は除外）。バックエンド: Go、`migrate up`、`go test`。Postgres（pgvector/pg17）、Redis（テスト用）。 |

詳細は `.github/workflows/ci.yml` を参照する。

## 6. リリースとデプロイ（プラットフォーム成果物）

プロダクトの顧客環境へデプロイする手順は案件ごとに異なる。ここでは **Multica 自体の配布**に限定する。

### 6.1 `main` とリリースの関係

- マージ先の正は `main`。

### 6.2 バージョンタグと GitHub Release

タグ `vX.Y.Z`（および `vX.Y.Z-<suffix>`）の push が [`.github/workflows/release.yml`](https://github.com/multica-ai/multica/blob/main/.github/workflows/release.yml) を起動する。成果物の概要: CLI（GoReleaser）、GHCR の `multica-backend` / `multica-web`、Desktop（Linux/Windows は CI 等）。macOS は手動経路が併存しうる。

### 6.3 セルフホスト利用者への反映

- `SELF_HOSTING.md` に従い、GHCR イメージとタグを対応づけてデプロイする。

### 6.4 Multica Cloud（マネージド）

クラウド側パイプラインは本リポに含まれない場合がある。プロダクション反映は内部 runbook に従い、**どの SHA / タグが走っているか**を追跡可能にする。

### 6.5 CLI とプロダクションの対応

プロダクション更新時は、利用者向け **CLI の整合**のため **パッチタグと GitHub Release** を推奨（`CLAUDE.md` の CLI 手順）。例外はリリースノートに理由を書く。

---

## 7. 検証環境の戦略（プロダクト × プラットフォーム）

| 対象 | 第1段の考え方 |
|------|----------------|
| **各プロダクト（顧客・案件）** | 共有または案件専用の **staging** を 1 系統用意し、本番に近い確認をそこで行う。 |
| **Multica 本体の変更** | チームが **Multica 用の共有 staging** を持つなら、高リスク PR のマージ前確認に使う。 |
| **拡張** | 滞留や UI 回帰が課題なら、大型・高リスクに限定して **PR プレビュー**や環境分離を追加する。 |

すべての PR に常設環境を用意する必要はない。**プロダクトの本番影響**が大きいほど、検証環境の線を引く。

---

## 8. ホットフィックス（プラットフォーム）

- 原則: PR 経由で `main` に戻し、必要なら **パッチタグ**で急いでリリースする。
- 例外: 直接コミットを許容する場合は、事後 **必ず PR で整形と説明**し、**どのプロダクト・利用者に効いたか**を記録する。

---

## 9. 変更・問い合わせ

- 本ポリシーの改訂は PR で行う。
- 「標準か高リスクか」迷うときは **高リスクに寄せる**か、オーナーに一度確認する。
