# Multica 運用戦略（プロダクト運用中心）

## 目的

Multica を「複数プロダクトの開発・保守・運用を回すプラットフォーム」として使う前提で、レビュー滞留を防ぎつつ品質を維持する。

## 基本方針

- レビュー工程は **機械 / AI / 人間** の3層で分業する。
- PR判断は **通常レーン / 高リスクレーン** に固定し、都度判断を減らす。
- 迷ったら高リスクに寄せる。

## レーン定義

### 通常レーン

- 対象: UI軽微、ドキュメント、テスト追加、小規模リファクタ。
- マージ条件:
  - CIグリーン
  - PR要約（自動生成可）
  - 承認1名

### 高リスクレーン

- 対象: 認証、DB/migration、runtime/daemon、本番設定、破壊的変更。
- マージ条件:
  - CIグリーン
  - 影響範囲/ロールバック記載
  - 追加検証結果
  - 承認2名（少なくとも1名は該当領域オーナー）

## 自動化の切り分け

### 機械化（100%自動）

- lint / typecheck / test / build / security scan
- PRテンプレ必須項目の入力チェック
- 変更ファイルによる自動ラベル（`risk:normal` / `risk:high`）

### 準自動（AI下書き）

- 差分要約
- 影響範囲の抽出
- テスト観点の提案
- レビュアー向け「重点確認ポイント」生成

### 人間のみ

- 仕様妥当性
- 顧客影響の最終判断
- Go / No-Go判断

## 日次・週次運用

- 毎日15-30分のレビュー専用枠を確保する。
- 進行中PRの上限を設ける（担当ごと）。
- 大PRは禁止し、1PR1意図を原則とする。
- 週次で滞留PRの原因を棚卸しする（レビュー待ち、検証待ち、仕様待ち）。

## 90日ロードマップ

### Phase 1（0-2週）

- レーン運用開始
- PRテンプレを通常/高リスクで運用開始
- 必須CIの固定

### Phase 2（3-6週）

- AI要約/レビュー観点の自動生成を標準化
- レビューSLA（通常: 当日、 高リスク: 翌営業日）を運用

### Phase 3（7-12週）

- 高リスク向け検証環境を強化（必要に応じてPRプレビュー導入）
- 失敗/手戻りパターンをテンプレへ反映

## KPI

- `in_review` 平均滞留時間
- PR作成からマージまでの中央値
- 差し戻し率（通常/高リスク別）
- リリース後の緊急修正率
- 1PRあたりの人手レビュー時間
